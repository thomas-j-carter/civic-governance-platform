# KEYCLOAK_ROLE_CLAIM_MAPPING

## Document Status
- Status: Canonical working baseline
- Purpose: define how Keycloak roles and claims map into Ardtire application roles and authorization behavior
- Audience: platform engineers, auth implementers, administrators, future AI implementation agents

---

## 1. Why This Document Exists

The platform now uses Keycloak as the external identity provider for browser sign-in and bearer-token verification.

That only solves authentication.

To build a coherent institution-grade authorization model, we must also define:
- which Keycloak roles exist
- where those roles live
- which claims the web app and gov-api read
- how those claims become Ardtire application roles
- how those application roles relate to permissions and institutional authority

Without a frozen mapping, identity integration drifts and becomes environment-specific folklore.

---

## 2. Core Principle

Keycloak is the identity provider.

Ardtire application roles are the authorization vocabulary used by:
- the web app
- the governance API
- later CMS/editorial access control
- future operational surfaces

The role model should be stable across clients.

---

## 3. Recommended Keycloak Topology

### Realm
Use a dedicated realm for the platform:
- `ardtire`

### Clients
Recommended clients:
- `ardtire-web` — browser sign-in / authorization-code flow for the web application
- `gov-api-service` — API/service audience and machine-oriented integration boundary
- `ardtire-cms` — dedicated editorial/admin client for CMS SSO when fully wired

The current web and gov-api implementation can operate before the CMS client is fully introduced, but the role model should already account for it.

---

## 4. Canonical Application Roles

The canonical application roles are:

- `authenticated_user`
- `applicant`
- `member`
- `reviewer`
- `editor`
- `officer`
- `certifier`
- `admin`
- `founder`

These roles are currently defined in application code and should also exist as Keycloak roles when they need to participate in token-based authorization.

---

## 5. Recommended Role Placement in Keycloak

### Realm Roles
Use realm roles for cross-application institutional roles that should be visible everywhere.

Recommended realm roles:
- `applicant`
- `member`
- `reviewer`
- `editor`
- `officer`
- `certifier`
- `admin`
- `founder`

### Client Roles
Use client roles only when behavior must be scoped to a specific client.

Examples:
- special CMS-only operational roles
- internal service roles
- temporary rollout roles

### Recommendation
Prefer realm roles for the canonical institutional roles above so:
- the web app can render route access consistently
- gov-api can enforce action permissions consistently
- CMS can later consume the same role model

---

## 6. Claim Sources the Application Reads

The current application-side mapping logic reads roles from:

- `realm_access.roles`
- `resource_access[client].roles`

The mapping layer then filters those values against the canonical application role set.

This means:
- extra roles may exist in Keycloak without automatically becoming app roles
- only explicitly recognized roles become Ardtire authorization roles

That filtering behavior is intentional.

---

## 7. Web Application Mapping

The web app should treat Keycloak claims as the source of authenticated user identity.

The web app currently maps:
- `sub`
- `email`
- `name`
- `preferred_username`
- recognized application roles

The web app uses those roles for:
- route protection
- UI visibility
- server-function authorization context forwarding

The web app must not invent local role truth that conflicts with Keycloak-issued claims.

---

## 8. Governance API Mapping

The governance API verifies bearer tokens and extracts:
- subject identifier
- email if present
- recognized application roles

The governance API then combines those roles with:
- application permission logic
- policy checks
- workflow state checks
- institutional object constraints

This means Keycloak roles are necessary but not sufficient.

A token role such as `reviewer` or `admin` enables broad authorization context, but governed actions still depend on:
- object state
- workflow legality
- policy conditions
- domain invariants

---

## 9. Role-to-Permission Intent

This document does not replace the application permission map, but the intent is:

### `applicant`
- may create and submit membership applications

### `member`
- may access member-only routes and future participation surfaces

### `reviewer`
- may review membership applications
- may move applications into review
- may return for revision

### `editor`
- intended for editorial/CMS access
- should not imply governance certification power

### `officer`
- may have proposal- or office-derived capabilities depending on later policy

### `certifier`
- may certify designated outcomes or record classes

### `admin`
- broad operational/admin authority
- still not a substitute for workflow legality

### `founder`
- bootstrap elevated authority during early phases
- should be progressively constrained as explicit governance roles mature

---

## 10. CMS Alignment Rule

The CMS/editorial surface must converge on the same identity source and role vocabulary.

That means:
- CMS editorial access should ultimately derive from Keycloak identity
- editor/admin access should map from the same canonical roles
- CMS must not become a separate shadow-identity system with divergent editorial roles

### Minimum expectation
- `editor` grants editorial content-management access
- `admin` grants elevated editorial/administrative access
- `founder` may be accepted during bootstrap, but not as a permanent substitute for real editorial/admin policy

---

## 11. Recommended Keycloak Configuration Notes

### For `ardtire-web`
- standard browser login client
- authorization code flow
- redirect URIs for local/staging/prod web hosts
- logout redirect URIs configured
- roles emitted in tokens via standard realm/client role mappers

### For `gov-api-service`
- configure audience expectations cleanly
- keep API validation aligned to expected issuer and accepted audiences
- do not rely on frontend-only role checks

### For future `ardtire-cms`
- separate client recommended
- same canonical realm roles
- dedicated redirect URIs for CMS host
- editorial/admin access derived from shared roles, not parallel local auth semantics

---

## 12. Token Mapping Safeguards

The application should continue to:
- ignore unknown roles
- normalize roles through a single mapping layer
- avoid stringly typed role logic scattered across apps
- treat tokens as role context, not as proof that any action is lawful

---

## 13. Bootstrap vs Mature State

### Bootstrap state
- founder/admin roles may be broader
- some users may be auto-provisioned into local persistence on first authenticated request
- development fallbacks may exist

### Mature state
- provisioning becomes deliberate and audited
- dev header fallbacks disabled outside local development
- all significant roles issued explicitly through Keycloak governance/ops process
- CMS/editorial access no longer depends on bootstrap assumptions

---

## 14. Source-of-Truth Rule

When there is conflict:
- Keycloak is authoritative for authenticated identity and issued roles
- application code is authoritative for role recognition, permission mapping, and workflow legality
- governance policy remains authoritative for what actions are actually valid

---

## 15. Summary

The correct long-term posture is:

- one external identity source: Keycloak
- one canonical role vocabulary: Ardtire application roles
- one mapping layer from token claims to recognized roles
- multiple application surfaces consuming the same role model
- governed actions still enforced by application and domain rules, not by token roles alone

# LOCAL_KEYCLOAK_SETUP

## Document Status
- Status: Working runbook
- Purpose: define the local-development Keycloak setup required for the current Ardtire auth integration
- Audience: developers, operators, future AI implementation agents

---

## 1. Goal

This runbook sets up local Keycloak so that:
- the web app can sign users in through OIDC
- the web app can store a server-side app session
- gov-api can verify bearer access tokens
- local role-driven member/admin route protection can be exercised without ad hoc fake identities

---

## 2. Local Host Assumptions

Expected local endpoints:
- Keycloak: `http://localhost:8080`
- Web app: `http://localhost:3000`
- Gov API: `http://localhost:3002`
- CMS: `http://localhost:3001`

Create realm:
- `ardtire`

Create these realm roles:

- `applicant`
- `member`
- `reviewer`
- `editor`
- `officer`
- `certifier`

Do not create ad hoc near-duplicates unless a deliberate ADR later introduces them.

### Future: `ardtire-cms`

Not required for the current web/gov-api flow, but recommended before full CMS SSO is completed.

The current stack expects these values:

### Shared auth
- `SESSION_SECRET`

### Keycloak
- `KEYCLOAK_BASE_URL`
- `KEYCLOAK_REALM`
- `KEYCLOAK_CLIENT_ID`

## 8. Recommended Local Flow

1. Start Keycloak.
2. Ensure realm, roles, clients, and local test users exist.
3. Start PostgreSQL.
4. Run Prisma generate/migrate/seed.
5. Start gov-api.
6. Start web.
7. Visit `http://localhost:3000/login`.
8. Sign in through Keycloak.
9. Verify:
   - member routes redirect correctly before sign-in
   - member routes open after sign-in
   - admin routes require reviewer/admin/founder role
   - membership flow functions through authenticated server calls

---

## 9. Local Provisioning Note

The current stack supports authenticated-user auto-provisioning into the local database for development convenience.

That means:
- a successfully authenticated user may be upserted on first request
- this is acceptable for bootstrap/local use
- this should become more explicit and policy-driven as the platform matures

---

## 10. What to Verify After Setup

### Web
- `/login` redirects to Keycloak
- `/auth/callback` completes sign-in
- header shows signed-in state
- `/logout` clears the session and returns cleanly

### Member surface
- unauthenticated access redirects to login
- authenticated applicant can create/submit membership applications
- approved membership appears on status page

### Admin surface
- unauthenticated access redirects to login
- non-admin/non-reviewer users cannot access admin routes
- reviewer/admin can process the queue

### Gov API
- bearer token verification succeeds
- invalid tokens are rejected
- dev header fallback can be disabled and the system still works with real sign-in

---

## 11. Known Current Boundary

At this stage:
- web and gov-api are aligned to Keycloak-based identity
- CMS/editorial identity alignment is documented and should follow the same role vocabulary
- full Payload-admin SSO replacement is a dedicated next step, not something to improvise partially

---

## 12. Summary

A correct local Keycloak setup gives the platform:
- real browser sign-in
- real bearer-token verification

- a stable foundation for removing remaining bootstrap-only auth behavior

apps/web/src/features/auth/route-access.ts

import type { AuthenticatedUser } from '../../lib/auth/session'

export type RouteRedirectTarget =
  | {
      to: '/login'
      search: { redirect: string }
    }
  | {
      to: '/'
    }

export function canAccessAdminSurface(
  user: AuthenticatedUser | null | undefined,
): boolean {
  if (!user) return false

  return user.roles.some((role) =>
    ['admin', 'reviewer', 'founder'].includes(role),
  )
}

export function getMemberRouteRedirectTarget(
  user: AuthenticatedUser | null | undefined,
  locationHref: string,
): RouteRedirectTarget | null {
  if (!user) {
    return {
      to: '/login',
      search: { redirect: locationHref },
    }
  }

  return null
}

export function getAdminRouteRedirectTarget(
  user: AuthenticatedUser | null | undefined,
  locationHref: string,
): RouteRedirectTarget | null {
  if (!user) {
    return {
      to: '/login',
      search: { redirect: locationHref },
    }
  }

  if (!canAccessAdminSurface(user)) {
    return {
      to: '/',
    }
  }

  return null
}

apps/web/src/features/auth/route-access.test.ts

import { describe, expect, test } from 'vitest'
import {
  canAccessAdminSurface,
  getAdminRouteRedirectTarget,
  getMemberRouteRedirectTarget,
} from './route-access'
import type { AuthenticatedUser } from '../../lib/auth/session'

function makeUser(overrides?: Partial<AuthenticatedUser>): AuthenticatedUser {
  return {
    sub: 'user_1',
    email: 'user@example.com',
    name: 'User Example',
    preferredUsername: 'user',
    roles: ['member'],
    ...overrides,
  }
}

describe('route access helpers', () => {
  test('member routes redirect unauthenticated users to login with redirect target', () => {
    expect(getMemberRouteRedirectTarget(null, '/member/applications')).toEqual({
      to: '/login',
      search: {
        redirect: '/member/applications',
      },
    })
  })

  test('member routes allow authenticated users through', () => {
    expect(
      getMemberRouteRedirectTarget(makeUser(), '/member/applications'),
    ).toBeNull()
  })

  test('admin routes redirect unauthenticated users to login', () => {
    expect(getAdminRouteRedirectTarget(null, '/admin')).toEqual({
      to: '/login',
      search: {
        redirect: '/admin',
      },
    })
  })

  test('admin routes redirect authenticated non-admin users home', () => {
    expect(getAdminRouteRedirectTarget(makeUser(), '/admin')).toEqual({
      to: '/',
    })
  })

  test('reviewer role grants admin surface access', () => {
    expect(
      canAccessAdminSurface(
        makeUser({
          roles: ['reviewer'],
        }),
      ),
    ).toBe(true)
  })

  test('admin role grants admin surface access', () => {
    expect(
      canAccessAdminSurface(
        makeUser({
          roles: ['admin'],
        }),
      ),
    ).toBe(true)
  })

  test('founder role grants admin surface access', () => {
    expect(
      canAccessAdminSurface(
        makeUser({
          roles: ['founder'],
        }),
      ),
    ).toBe(true)
  })

  test('plain member role does not grant admin surface access', () => {
    expect(
      canAccessAdminSurface(
        makeUser({
          roles: ['member'],
        }),
      ),
    ).toBe(false)
  })
})

What this batch accomplishes

This batch locks down three important things:

the Keycloak role model is now documented instead of implicit

local auth setup is now reproducible instead of remembered

member/admin route protection is now driven by a tested helper instead of duplicated inline logic


One honest boundary

I did not fully replace Payload’s internal admin auth in this batch. The right way to do that is a dedicated CMS SSO bridge against Keycloak, not a rushed half-integration. What is now frozen is the important part: CMS must converge on the same Keycloak identity source and the same canonical role vocabulary (editor, admin, founder, etc.) rather than inventing a parallel role system.

Best next batch

The strongest next move is the dedicated CMS SSO/auth convergence batch:

introduce a Keycloak client for apps/cms

wire CMS login/callback/logout routes

map Keycloak editor/admin roles into editorial access control

remove reliance on Payload-local editorial identity for primary admin access
